# Работа программы

- При запуске программы появляется приветственная строка
  
  ```
  Начать новую игру или выйти? (newgame/exit)
  ```
  
  Вызывает диссонанс что это единственное место где используется латиница, хотя весь остальной интерфейс программы на кириллице. Лучше соблюдать единую принятую конвенцию для интерфейса.

- При появлении интерфейса самой игры смущает следующая строка
  
  ```
  Всего ошибок: 0. Последний ввод: -.
  ```
  
  То что у пользователя 0 ошибок это радует, но мне кажется лучше вообще это не выводить пока ошибок нет)
  Да и само кол-во совершенных ошибок мало о чем говорит.

В качестве предложения два варианта:

1. Выводить эту часть интерфейса только после совершения первой ошибки и в формате
   
   ```
   Всего ошибок: 1/7
   ```
2. Или вообще выводить изначально только кол-во оставшихся попыток
   
   ```
   Осталось попыток: 7
   ```

Вот эта часть тоже смущает ***Последний ввод: -.***
Создается впечатление что у пользователя был последний ввод '-.', что создает замешательство.

Да и вообще, сомнительна сама потребность в такой информации.
Лучше выводить использованные пользователем буквы или которые отсутствуют в слове.


Что-то вроде такого:

```
Использованные буквы: а и о е
```

И мне кажется что лучше эти две "панели" разнести по разным строчкам, потому-что у них разный контекст информирования пользователя.

Вместо:

```
Осталось попыток: 7. Использованые буквы: а и о е.
```

Лучше так:

```
Осталось попыток: 7
Использованые буквы: а и о е
```

Можешь при желании как-то отцентрировать или отформатировать, но это не особо важно.

- Паузы между переходами

Они сбивают с толку, не сразу понимаешь что произошло да и не всегда успеваешь прочитать последнее сообщение.
Например пока я читал поздравление с победой/проигрышем, экран очистился и запустилась новая сессия.

Лучше просто сделай два "экрана" для этих состояний.
Один для победы и один для поражения.
И уже там предлагай продолжить/завершить игру.

- В остальном программа работает корректно и соответствует функционалу описанному в ТЗ.

## init_newgame_or_exit(words, hangman_stages):

Старайся абстрагировать функции таким образом, чтобы они соблюдали принцип единой ответственности и отвечали только за что-то одно.

Тут у тебя в названии инициация игры или выход. Это сбивает с толку.
И по логике информация о hangman_stages должна быть ответственностью самой игры, а не загружаться в нее через внешний параметр. Т.е. перенеси чтение файла **<u>hangman_stages.json</u>** внутрь игровой логики, например в ту же функцию <u>**init_newgame_or_exit**</u>.


## game_cycle(true_word, mistakes_count, hidden_word, used_letters, hangman_stages):

😳 Ого...
Не находишь что тут слишком много параметров пробрасываются в функцию извне, хотя по логике они должны создаваться уже внутри функции?

Вот место где ты вызываешь эту функцию

```python
true_word: str = choice(words)
mistakes_count: int = 0
hidden_word: list[str] = ["_"] * len(true_word)
used_letters: list[str] = []
game_cycle(true_word, mistakes_count, hidden_word, used_letters, hangman_stages)
```

И единственное что из этого имеет смысл пробрасывать как некий внешний объект, от которого зависит и меняется работа самой функции, это **true_word**.

**true_word** кстати не лучшее имя для этой сущности, сгодится и просто **word**

Остальные сущности у тебя несут роль "пустого объекта" и создавать их должна уже сама игра.


## check_game_over(mistakes_count, true_word, hidden_word, max_mistakes) -> bool:

Такие функции лучше обрамлять префиксом **is**, что подчеркивает их предикативность.

Здесь тоже много параметров, но здесь это уместно, потому-что все они нужны для работы функции и кроме как извне, она получить их не может.

Внутри самой функции ты вызываешь **announce_result** которая запускается если игра окончена, очищает экран и выводит сообщение в зависимости от результата. Это очень плохо.

1. Это нарушает ответственноть функции **check_game_over**, она должна ТОЛЬКО проверять, окончена игра или нет и больше НИЧЕГО.
2. Создает не очевидное поведение с побочным эффектом очистки экрана и вывода сообщения.

Такие функции трудно тестировать и дебажить.
Сведи ее до максимально простой.

```python
def check_game_over(mistakes_count: int, true_word: str, hidden_word: list[str], max_mistakes: int) -> bool:
    return is_user_guess_word(true_word, hidden_word) or is_user_lose(mistakes_count, max_mistakes)
```

В дополнение можешь создать еще пару предикативных функций:

```python
def is_user_guess_word(true_word: str, hidden_word: list[str]) -> bool
    return true_world == "".join(hidden_word)

def is_user_lose(mistakes_count: int, max_mistakes: int) -> bool
    return mistakes_count >= max_mistakes
```

И по результату функции **check_game_over** определять логику внутри функции **game_cycle**, чтобы не вызывать внутри нее **announce_result**.

Вот примерно то что я имею в виду.

```python
# game_cycle
max_mistakes = len(hangman_stages) - 1

while not check_game_over(mistakes_count, true_word, hidden_word, max_mistakes):

# сюда попадаем когда игра окончена
if is_player_lose(mistakes_count, max_mistakes):
    announce_result(win=False, true_word=true_word)
elif is_player_guess_word():
    announce_result(win=True, true_word=true_word)
else:
    # какая-то ошибка, такого не должно быть
```
